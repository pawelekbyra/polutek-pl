# LANE-video-provider

## 0. Purpose

This lane owns the provider-agnostic video foundation for Polutek.pl.

It exists to make sure that:

```txt id="r6akeq"
video provider integration is thin
Mux and Cloudflare Stream can be supported as candidates
R2/S3 remain legacy/migration storage only
upload goes direct browser -> provider
provider webhooks update internal VideoAsset state
provider-specific IDs live in VideoAsset, not Video
provider playback source is created only after app access allows it
```

This lane must not become a giant multi-provider framework.

Core invariant:

```txt id="yy8xse"
Access decision belongs to the application, not to Mux, Cloudflare, R2, or S3.
```

## 2. Product and architecture rules

### 2.1 Provider is not access

Video providers may store, process, and serve video.

They must not decide application access.

Correct flow:

```txt id="d3btb2"
User asks for playback
  -> app checks Access / PatronGrant
  -> app checks primary READY VideoAsset
  -> if allowed, app asks provider for signed playback source
  -> app returns PlaybackPlan
  -> player renders
```

Locked flow:

```txt id="uuftx9"
User asks for playback
  -> app checks Access / PatronGrant
  -> app decides denied
  -> app does not call provider
  -> app returns PlaybackPlan allowed=false
  -> UI renders locked placeholder
```

Forbidden:

```txt id="afpc8e"
provider call before access allow
provider source for denied user
provider token for locked content
provider-specific public IDs leaking as access substitute
raw private media URL as security model
```

### 2.2 Thin provider abstraction

Target:

```txt id="8lbgph"
thin VideoProvider interface
```

Not:

```txt id="ck68f2"
giant multi-provider framework
provider chaos
multi-tenant provider orchestration
active R2/S3 fallback playback
```

Minimal provider API may include:

```txt id="fvjwyh"
createUploadSession()
handleWebhook()
createPlaybackSource()
```

### 2.3 Candidate providers

Initial candidates:

```txt id="3tq2nb"
MUX
CLOUDFLARE_STREAM
```

Legacy/migration storage only:

```txt id="oh3wgc"
R2_LEGACY
S3_LEGACY
```

R2/S3 must not become active fallback playback on start.

### 2.4 Video vs VideoAsset

Target separation:

```txt id="hwtfqk"
Video = content metadata and product state
VideoAsset = provider/media processing/playback state
```

`Video` owns:

```txt id="5lqc91"
title
description
slug
tier
publication status
sidebar order
featured state
admin CRUD metadata
```

`VideoAsset` owns:

```txt id="ixdeqh"
provider
provider-specific IDs
upload status
processing status
playback ID/source metadata
thumbnail/poster metadata if used
duration
primary/fallback role
provider webhook state
```

### 2.5 Primary READY asset

Runtime should use:

```txt id="0au0np"
primary READY VideoAsset
```

Not:

```txt id="fv4yb1"
random provider asset
old raw URL
fallback R2 link
Video.providerPlaybackId directly from Video if target model moved it to VideoAsset
```

## 3. Domain concepts

### VideoProvider

A thin interface for provider-specific actions.

Potential operations:

```txt id="y94yfb"
createUploadSession(input)
parseWebhook(rawRequest)
handleWebhookEvent(event)
createPlaybackSource(input)
getAssetStatus(input)
deleteAsset(input)
```

Keep it thin.

Do not abstract the entire provider universe.

### VideoAsset

Internal media asset record.

Potential fields:

```txt id="e3suup"
id
videoId
provider
role
isPrimary
status
providerAssetId
providerPlaybackId
providerStreamUid
durationSeconds
thumbnailUrl
metadata
createdAt
updatedAt
```

Potential statuses:

```txt id="7xlarj"
UPLOAD_SESSION_CREATED
UPLOADING
PROCESSING
READY
ERROR
DELETED
```

Potential roles:

```txt id="u8sq0j"
PRIMARY
FALLBACK
TRAILER
LEGACY
```

### Upload session

Upload should go:

```txt id="zw5npa"
admin browser -> provider
```

not:

```txt id="25slkz"
admin browser -> app backend -> provider
```

The app backend should create a provider upload session, not proxy huge files.

### Provider webhook

Provider webhooks update internal state.

Example:

```txt id="lcqgt2"
Mux webhook
  -> verify/parse Mux event
  -> map to internal VideoAsset event
  -> update VideoAsset via use case

Cloudflare webhook
  -> verify/parse Cloudflare event
  -> map to internal VideoAsset event
  -> update VideoAsset via use case
```

## 14. Final lane rule

This lane is successful when Playback/Player can later say:

```txt id="f5mgs1"
I do not care whether the video is on Mux or Cloudflare.
I receive a PlaybackPlan.
If allowed, it contains a safe playback source.
If denied, it contains no source and no token.
```

That is the handoff from Video Provider Foundation to PlaybackPlan / Player.
