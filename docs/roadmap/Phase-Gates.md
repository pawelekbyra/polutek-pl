# Phase Gates — active

Status: ACTIVE — POST-R AI DELIVERY CONTROL PLANE.

No phase can be certified by a Builder. Only Certifier can recommend certification. Human owner merges.

Every phase must record: entry criteria, exit criteria, validation required, certification required, docs reconciliation required, ticket movement required and blocker recording required.


## X0 — Control Plane & Truth Reconciliation

### Entry criteria

- R-phase handoff/previous phase is certified or owner explicitly approves entry.
- Active tickets exist with allowed/forbidden paths and validation.
- Known blockers are recorded.

### Exit criteria

- Required scope for X0 is merged or explicitly deferred with owner approval.
- No launch-critical blocker remains unrecorded.
- Docs and current main are reconciled.

### Validation required

- Ticket-specific validation.
- `git diff --check` for docs/PR hygiene.
- Architecture guard/typecheck/tests when runtime changes occur in active future tickets.
- Manual QA where domain requires it.

### Certification required

- Certifier report in `docs/reports/certification/`.
- Verdict recommends `CERTIFY`, `FIX`, or `BLOCKED`.
- Owner makes final merge/status decision.

### Docs reconciliation required

- Roadmap, owner timeline, relevant spec/lane and ticket status updated by Integrator.
- Target docs must not pretend future runtime is current implementation.

### Ticket movement required

- Completed tickets move to `done/` after merge/reconciliation.
- Blocked tickets move to `blocked/` with unblock condition.
- New follow-ups go to `ready/` only after Planner/owner approval.

### Blocker recording required

- Every blocker must state owner decision needed, technical dependency, validation gap or merge conflict.
- X7 blockers must say whether they block public launch.

## X6 — Product Excellence Passes

X6 is a measurable Product Excellence program split into separate passes. It is not one broad rewrite, not launch certification and not proof of current runtime. Every pass uses the shared statuses: `IMPLEMENTED_VERIFIED`, `IMPLEMENTED_UNVERIFIED`, `PARTIAL`, `MISSING`, `BLOCKED`, `OWNER_DECISION_REQUIRED`, `DEFERRED_POST_LAUNCH`, `NOT_APPLICABLE`.

### Entry criteria

- Previous phase is certified or owner explicitly approves entry.
- Active X6 ticket exists with allowed/forbidden paths and validation.
- Launch-critical screens/flows touched by the pass are named.
- Known blockers and owner questions are recorded.
- The ticket identifies automated evidence, manual evidence and production/prod-like evidence required.

### Required X6 pass structure

Every X6 pass MUST state: goal, scope, entry criteria, evidence required, measurable exit criteria, launch blockers, should-have, post-launch items, owner questions and candidate small tickets. Evidence MUST use this matrix:

## X7 — Launch Readiness / Final Certification

X7 is Launch Proof, not a checklist. X7 requires a complete Launch Evidence Pack proving production readiness. Green preview deployment, local tests, docs or a `MERGE` recommendation are not enough.

### Entry criteria

- Previous phase is certified or owner explicitly approves entry.
- Active X7 ticket exists with allowed/forbidden paths and validation.
- X6 evidence gaps, deferrals and owner-approved exceptions are listed.
- Production environment, credentials/access, runbook and owner decision register are ready for evidence collection.

### Launch Evidence Pack required

| Area | Required proof |
| --- | --- |
| Deployment | Green production deployment, green build from current main, production domain/HTTPS, verified production env groups, no secrets in repo/public runtime, provider dashboards pointing at production URL, runtime logs checked, rollback documented, no reliance on green preview instead of production. |
| Payment and access | Qualifying payment -> verified Stripe webhook -> Payment fact -> eligibility policy -> active PatronGrant -> access allow; negative cases for below-threshold, duplicate webhook, payment alone, Clerk metadata, `User.isPatron`, newsletter subscription, guest and non-patron; lifecycle for full refund, dispute opened/won/lost-chargeback, manual grant/suspend/reactivate/revoke with reason/audit. |
| Video and playback | Admin upload/import, Cloudflare lifecycle to READY, primary READY asset, allowed patron playback, denied guest/non-patron, no playback URL/token/provider call/player mount/session/view for denied, no legacy private fallback for patron-only content, PROCESSING/FAILED/NO_PRIMARY_ASSET states, mobile/desktop player proof. |
| Comments | Public read of published comments, guest cannot write/react/report, non-patron cannot write on patron-only, patron can write per policy, admin moderation, report abuse, rate limit, duplicate/spam protection, hide/delete/restore/dismiss audit, no shadow bans. |

| Security | Risk-based review using OWASP ASVS 5.0.0 for launch-critical authentication, session, authorization, input validation, webhook verification, payment integrity, upload/import, SSRF/URL import risk, secrets, logging, error handling, dependency risk, admin actions, rate limiting, privacy and token/source leakage; exclusions justified. |
| Accessibility, mobile and performance | Final evidence that launch-critical flows meet target or have explicit blocker/accepted exception; mobile/browser matrix run; performance budgets checked; no critical layout/interaction/navigation problem; owner accepted recorded exceptions. |
| Legal and owner | Privacy policy, terms, cookie/consent behavior, email consent, support/contact path, refund/dispute communication, material retention policy, Cloudflare cost/retention policy, resolved owner questions and no agent-made owner decisions. |

### Exit criteria

- Launch Evidence Pack has status/evidence/environment/date/owner for every required area.
- Missing required proof is not hidden as pass; it is `IMPLEMENTED_UNVERIFIED`, `PARTIAL`, `MISSING` or `BLOCKED`.
- No launch-critical blocker remains unrecorded.
- Final certification report states exactly one of `GO`, `CONDITIONAL_GO`, `NO_GO`.
- Docs and current main are reconciled without claiming launch-ready from docs alone.

### Final GO / NO-GO semantics

- `GO`: no launch-critical blockers and all required evidence is present.
- `CONDITIONAL_GO`: allowed only for explicit non-critical exceptions accepted by owner. It MUST NOT be used for access, payments, token leakage, legal/privacy, broken production deploy or missing backup/recovery proof.
- `NO_GO`: at least one launch-critical blocker exists or mandatory evidence is missing.
